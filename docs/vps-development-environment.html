Ubuntu VPS Development Environment

Rebuild and operate a persistent Ubuntu 24.04 amd64 development VPS reachable only over Tailscale from Linux, Mac, and iPhone. Private keys stay on clients; the server hosts the outer tmux workspace, inner Herdr agent UI, and a manual Metro window for VerseGuard.

1. Document scope

Generated / scope note This standalone HTML documents the VPS session architecture and the current repository CLI. It is not a dump of live host state. Commands use placeholders such as YOUR_VPS_USER. Do not paste public IPs, Tailscale MagicDNS hostnames, OAuth codes, fingerprints, or personal usernames into committed copies of this file.
  • In scope: rebuild order, Tailscale-only access, installer modes, shell/editor/agents, tmux/Herdr, AppArmor sandbox, client terminfo, VerseGuard Metro over Tailscale Serve, validation and troubleshooting.
  • Out of scope: provider account creation, DNS for the public internet, Docker/Lazydocker, automatic Metro systemd enablement (unit is optional and intentionally manual).
  • Security posture: the final state has no public SSH exposure and never exposes Metro beyond the tailnet. Keep any provider-console or temporary provisioning access only until Tailscale is verified.

4. Ubuntu rebuild, sudo user, SSH hardening

Goal: a non-root daily user, key-only SSH, and a lockout-safe hardening order. Keep one console/VNC or temporary public SSH session open until Tailscale access is proven.

4.1 Create the sudo user (as root / recovery console)

VPS · root/console
adduser YOUR_VPS_USER
usermod -aG sudo YOUR_VPS_USER
mkdir -p /home/YOUR_VPS_USER/.ssh
chmod 700 /home/YOUR_VPS_USER/.ssh
# paste the client's public key only:
nano /home/YOUR_VPS_USER/.ssh/authorized_keys
chmod 600 /home/YOUR_VPS_USER/.ssh/authorized_keys
chown -R YOUR_VPS_USER:YOUR_VPS_USER /home/YOUR_VPS_USER/.ssh

Why: daily work must not run as root. Public keys authenticate clients; private keys never leave the client.

4.2 Generate / register keys on each client

Client · Linux/Mac
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_vps -C "YOUR_CLIENT_USER@client-to-vps"
# Copy only the .pub line into the VPS authorized_keys file.
# Keep the private key on this client (or in the client OS keychain / Moshi key store).

Why private keys are not “shared” with the VPS: SSH proves possession of the private key via challenge-response. The VPS only stores the matching public key.

4.3 Lockout-safe SSH hardening sequence

  1. Confirm you can log in as YOUR_VPS_USER with the new key (even if temporarily over public IP).
  2. Install and enroll Tailscale (next section) and confirm ssh YOUR_VPS_USER@YOUR_VPS_TAILSCALE_HOST.
  3. Restrict UFW so SSH is allowed only on tailscale0.
  4. Only then disable password/root login and remove any temporary public SSH exposure.
VPS · after Tailscale login works
sudo tee /etc/ssh/sshd_config.d/99-hardening.conf >/dev/null <<'EOF'
PermitRootLogin no
PasswordAuthentication no
KbdInteractiveAuthentication no
PubkeyAuthentication yes
EOF
sudo sshd -t && sudo systemctl reload ssh
Do not Do not disable password/root login before you have a working Tailscale SSH session from a second device. Keep a recovery path until verification succeeds.

5. Tailscale, UFW, Mosh, Expo ports

5.1 Install & enroll

VPS
# Follow current Tailscale Ubuntu install docs, then:
sudo tailscale up
tailscale status
tailscale ip -4
# MagicDNS name should resolve as YOUR_VPS_TAILSCALE_HOST.YOUR_TAILNET_DOMAIN

Why: MagicDNS gives stable hostnames inside the tailnet without publishing DNS to the internet.

5.2 UFW — allow services only on tailscale0

VPS · keep an active session
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow in on tailscale0 to any port 22 proto tcp comment 'SSH via Tailscale'
sudo ufw allow in on tailscale0 to any port 60000:61000 proto udp comment 'Mosh via Tailscale'
sudo ufw allow in on tailscale0 to any port 8081 proto tcp comment 'Expo Metro via Tailscale (optional direct)'
# Prefer Tailscale Serve for HTTPS to iOS; keep 8081 private to the tailnet if opened at all.
sudo ufw enable
sudo ufw status verbose
Verify before removing public SSH From a client already on the tailnet: tailscale ping YOUR_VPS_TAILSCALE_HOST, then ssh YOUR_VPS_USER@YOUR_VPS_TAILSCALE_HOST and/or Mosh. Only after that succeed, remove temporary public 22/tcp allow rules if any existed.

5.3 Client SSH config (optional convenience)

Client ~/.ssh/config
Host YOUR_VPS_TAILSCALE_HOST
  HostName YOUR_VPS_TAILSCALE_HOST.YOUR_TAILNET_DOMAIN
  User YOUR_VPS_USER
  IdentityFile ~/.ssh/id_ed25519_vps
  IdentitiesOnly yes

5.4 Mosh

Mosh is installed by bootstrap apt packages on the VPS. On clients, install Mosh separately. UDP 60000:61000 must be open on tailscale0 only.

Client
mosh YOUR_VPS_USER@YOUR_VPS_TAILSCALE_HOST.YOUR_TAILNET_DOMAIN -- tmux attach -t main

7. Shell: Zsh, Oh My Zsh, P10k, mise, Node

Server profile safe-links zsh/.zshenv → ~/.zshenv and zsh/.zshrc → ~/.zshrc. .zshenv owns portable PATH for both interactive and non-interactive Zsh (for example ssh host 'cmd').

Zsh scalar path warning In Zsh, path is a special array tied to PATH. Assigning a scalar string to path (for example path=/something) corrupts PATH. Always use array form: typeset -U path and path=(... $path), then export PATH. If commands “vanish” after editing shell files, inspect echo $PATH and typeset -p path first.
VPS · sanity
zsh -lic 'command -v mise; command -v node; command -v pi; command -v codex; echo $PATH'
# Expect ~/.local/bin, ~/.opencode/bin, ~/.local/opt/npm-ai/bin,
# and mise shims early in PATH.

10. Codex bubblewrap / AppArmor

Ubuntu can keep kernel.apparmor_restrict_unprivileged_userns=1. Codex’s sandbox needs bubblewrap; without a restricted AppArmor profile for /usr/bin/bwrap, sandboxed runs fail. The tracked profile allows bwrap to operate under that global restriction without turning the restriction off.

VPS
./install.sh server sandbox --dry-run
./install.sh server sandbox --apply
# Confirmation phrase: APPLY SANDBOX
cat /proc/sys/kernel/apparmor_restrict_unprivileged_userns   # must remain 1
/usr/bin/bwrap --unshare-user --uid 65534 --gid 65534 --ro-bind / / --dev /dev --proc /proc -- /bin/true
Profile scope This AppArmor profile attaches to /usr/bin/bwrap, so it affects every user invoking that executable—not only Codex. Review the tracked profile before applying it.
  • Profile source: server/apparmor/bwrap-userns-restrict → /etc/apparmor.d/bwrap-userns-restrict.
  • Dry-run is a no-op if the destination already matches; --apply reloads the identical profile to verify it is active and refuses to overwrite a differing foreign profile.
  • Codex tmux popup uses -s workspace-write -a on-request (see tmux/scripts/codex_toggle.sh), not a full sandbox bypass.

11. Client terminals: Ghostty, macOS, Moshi

11.1 Ghostty terminfo (client → server recipe)

If the client sets TERM=xterm-ghostty (or ghostty) but the VPS lacks that terminfo entry, colors collapse to gray, Alt/meta chords misbehave, and keys can appear to repeat. Install terminfo on the server from the client’s known-good entry, or fix TERM on the client.

Preferred · from Ghostty client
# On the Ghostty client, export terminfo to the VPS:
infocmp -x xterm-ghostty | ssh YOUR_VPS_TAILSCALE_HOST tic -x -o ~/.terminfo -
# Verify on VPS:
ssh YOUR_VPS_TAILSCALE_HOST 'infocmp xterm-ghostty | head'
Fallback
# One connection with a broadly available TERM value:
TERM=xterm-256color ssh -t YOUR_VPS_TAILSCALE_HOST
# Mosh fallback:
TERM=xterm-256color mosh YOUR_VPS_TAILSCALE_HOST
# Tradeoff: loses some Ghostty-specific keys/capabilities.

Doctor warns (does not fail) when Ghostty terminfo is missing on the server.

11.2 macOS Ctrl+Space conflict

Herdr’s prefix is Ctrl+Space. macOS often binds that to “Select previous input source”. Disable or rebind the OS shortcut (System Settings → Keyboard → Keyboard Shortcuts → Input Sources) so Herdr receives the chord inside the terminal.

11.3 Moshi / iPhone

  1. Install Tailscale on the phone; join the same tailnet.
  2. Generate or import an SSH key in Moshi; add only the public key to VPS authorized_keys.
  3. Host: YOUR_VPS_TAILSCALE_HOST.YOUR_TAILNET_DOMAIN, user YOUR_VPS_USER.
  4. Transport: Auto or Mosh; remote command or post-login: attach tmux main.

VPS · tmux window named metro

Create/rename a tmux window to metro so every client finds the same pane. Keep Metro bound to localhost; publish HTTPS through Tailscale Serve, not the public internet.

13.3 Tailscale Serve + iOS ATS

VPS
# HTTPS front for local Metro (port may be 8081 — confirm Expo's listening port):
sudo tailscale serve --bg https / http://127.0.0.1:8081
tailscale serve status
# Point the iOS dev client at YOUR_EXPO_HTTPS_URL (HTTPS, not http://100.x.y.z).
ATS rationale iOS App Transport Security blocks cleartext HTTP to Tailscale IP literals. Serve provides a trusted HTTPS URL on the tailnet so the Expo dev client can load the bundle.

13.4 Fast Refresh vs cold bundle

  • Cold: first bundle can take on the order of a minute for a large app; progress may show “Downloading…”.
  • Warm: Fast Refresh and cached bundles are typically sub-second.
  • Intermittent re-download behavior is usually Metro/dev-client cache lifecycle, not Tailscale failure — check Metro logs in the metro window.

16. Maintenance, rollback, rebuild checklist

16.1 Updates

  • Bump pins in server/versions.env with new URLs and sha256 digests, then re-run ./install.sh server bootstrap --yes (idempotent skips when versions match).
  • Re-run ./install.sh server doctor after upgrades.
  • Re-run sandbox apply after profile changes or to reload/verify the tracked profile after parser issues.

16.2 Rollback

Not transactional Bootstrap is idempotent but does not maintain an automatic rollback journal. Snapshot the VPS before major upgrades.
  • For dotfile configuration, check out the last known-good Git revision and re-run the safe link profile.
  • For pinned binaries, restore a VPS snapshot or review the previous manifest and downgrade deliberately; do not assume every package manager can roll back.
  • Safe-link mode refuses to clobber unknown files—rename conflicts intentionally after inspecting them.
  • Do not delete live auth directories or private runtime state when rolling back tracked configuration.

16.3 What not to commit

  • Filled placeholders, public IPs, MagicDNS hostnames, OAuth material, auth.json, private keys.
  • ~/.config/verseguard-metro.env and any env files with Serve URLs if treated as sensitive in your threat model.
  • Runtime state under Herdr / Pi that is gitignored by design.

16.4 Optional future work

  • Enable verseguard-metro.service if you want Metro across reboots without tmux.
  • Bundle optimization (tighter icon/font imports, lazy data) to shrink cold Metro times.
  • Docker + Lazydocker only if container workflows become required.

16.5 Concise rebuild checklist

  1. Provision Ubuntu 24.04 amd64; create YOUR_VPS_USER; install client public keys.
  2. Enroll Tailscale; verify MagicDNS; UFW allow SSH/Mosh/(optional 8081) on tailscale0 only.
  3. Harden SSH; remove public SSH after Tailscale proof.
  4. Clone dotfiles → ./install.sh server bootstrap --dry-run → --yes.
  5. ./install.sh server sandbox --apply; first interactive nvim; agent OAuth via ssh -L as needed.
  6. Create tmux main; start Herdr; fix Ghostty terminfo / macOS Ctrl+Space on clients.
  7. Clone VerseGuard; npm ci; Metro in tmux; Tailscale Serve HTTPS; point iOS at YOUR_EXPO_HTTPS_URL.
  8. ./install.sh server doctor; attach from Linux, Mac, and iPhone.
